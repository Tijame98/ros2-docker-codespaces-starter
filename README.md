# ROS2 Docker Codespaces Starter

A practical starter project to learn how to run ROS2 inside Docker and GitHub Codespaces.

The goal is to build a reproducible ROS2 development environment that works in the cloud, without requiring a powerful local PC, a local Ubuntu installation, or a local Docker setup.

This project starts with a basic ROS2 workspace, then extends it into a multi-container architecture using Docker Compose.

---

## Why this project?

Setting up ROS2 locally can be heavy, especially if your machine is not powerful or if you do not want to install Ubuntu, Docker, ROS2, and all dependencies manually.

This project solves that by using:

- GitHub Codespaces for cloud-based development
- Docker DevContainers for a reproducible ROS2 environment
- ROS2 Humble as the robotics middleware
- Docker Compose to run multiple ROS2 nodes in separate containers

It is designed as a base for future robotics projects such as simulation, navigation, perception, monitoring, and robot deployment.

---

## What you will build

This project contains two main versions.

### V1 — ROS2 inside a DevContainer

In V1, ROS2 runs directly inside a GitHub Codespaces DevContainer.

You will create and run a custom ROS2 package called:

```text
robot_status_demo
```

This package contains:

- a publisher node
- a subscriber node
- a launch file

The publisher sends messages on:

```text
/robot/status
```

The subscriber receives and prints these messages.

---

### V2 — ROS2 multi-container setup with Docker Compose

In V2, the same ROS2 nodes run in separate Docker containers.

Architecture:

```text
status_publisher container  --->  /robot/status  --->  status_subscriber container
```

This shows how ROS2 nodes can communicate across containers using Docker Compose.

# Getting Started

## 1. Open the project in GitHub Codespaces

From the GitHub repository page:

```text
Code → Codespaces → Create codespace on main
```

Codespaces will start a cloud VS Code environment.

The DevContainer will install and configure ROS2 Humble automatically.

---

## 2. Check ROS2 installation

Inside the Codespaces terminal, run:

```bash
printenv ROS_DISTRO
```

Expected output:

```text
humble
```

You can also check that the ROS2 command is available:

```bash
ros2
```

---

## 3. Test ROS2 with demo nodes

If the demo packages are installed, open two terminals.

Terminal 1:

```bash
ros2 run demo_nodes_cpp talker
```

Terminal 2:

```bash
ros2 run demo_nodes_py listener
```

Expected behavior:

```text
Publishing: 'Hello World...'
I heard: 'Hello World...'
```

If the demo packages are missing, install them with:

```bash
apt-get update
apt-get install -y ros-humble-demo-nodes-cpp ros-humble-demo-nodes-py
source /opt/ros/humble/setup.bash
```

# V1 — Run the custom ROS2 package

## 1. Build the workspace

From the project root:

```bash
cd ros2_ws
colcon build
source install/setup.bash
```

---

## 2. Run the publisher

Terminal 1:

```bash
ros2 run robot_status_demo status_publisher
```

Expected output:

```text
Publishing: Robot is alive from Dockerized ROS2 environment | count=0
```

---

## 3. Run the subscriber

Terminal 2:

```bash
source /workspaces/ros2-docker-codespaces-starter/ros2_ws/install/setup.bash
ros2 run robot_status_demo status_subscriber
```

Expected output:

```text
Received: Robot is alive from Dockerized ROS2 environment | count=0
```

---

## 4. Run both nodes with a launch file

Instead of opening two terminals, launch both nodes with:

```bash
ros2 launch robot_status_demo status_demo.launch.py
```

Expected behavior:

```text
status_publisher publishes messages
status_subscriber receives messages
```

# V2 — Run ROS2 nodes with Docker Compose

V2 runs the publisher and subscriber in separate containers.

This is useful because real robotic systems are often split into multiple services: perception, control, navigation, monitoring, communication, etc.

---

## 1. Check Docker availability

Run:

```bash
docker --version
docker compose version
```

If Docker is not found, make sure the DevContainer includes Docker-in-Docker in `.devcontainer/devcontainer.json`:

```json
"features": {
  "ghcr.io/devcontainers/features/docker-in-docker:2": {}
}
```

Then rebuild the container:

```text
Ctrl + Shift + P → Codespaces: Rebuild Container
```

---

## 2. Build and start the containers

From the project root:

```bash
docker compose up --build
```

Expected output:

```text
status_publisher   | Publishing: Robot is alive from Dockerized ROS2 environment | count=0
status_subscriber  | Received: Robot is alive from Dockerized ROS2 environment | count=0
```

---

## 3. Stop the containers

Use:

```bash
Ctrl + C
```

Then clean the containers:

```bash
docker compose down
```

---

## 4. Inspect ROS2 topics inside a container

While the containers are running, open another terminal and enter the publisher container:

```bash
docker exec -it status_publisher bash
```

Inside the container:

```bash
source /opt/ros/humble/setup.bash
source /ros2_ws/install/setup.bash
ros2 topic list
```

Expected output:

```text
/robot/status
/parameter_events
/rosout
```

You can also listen to the topic:

```bash
ros2 topic echo /robot/status
```

---

# ROS2 Communication Note

Both containers use the same ROS2 domain:

```yaml
ROS_DOMAIN_ID=42
```

This is defined in `docker-compose.yml`.

ROS2 uses DDS for communication. The `ROS_DOMAIN_ID` helps isolate ROS2 systems from each other. If publisher and subscriber are not communicating, one of the first things to check is whether both nodes use the same `ROS_DOMAIN_ID`.

# Common Issues

## Package not found

If you see:

```text
Package 'robot_status_demo' not found
```

Run:

```bash
cd ros2_ws
colcon build
source install/setup.bash
```

---

## Python module not found

If you see:

```text
ModuleNotFoundError: No module named 'robot_status_demo.status_publisher'
```

Check that the file exists here:

```text
ros2_ws/src/robot_status_demo/robot_status_demo/status_publisher.py
```

Then rebuild:

```bash
cd ros2_ws
rm -rf build install log
colcon build
source install/setup.bash
```

---

## Docker command not found

If you see:

```text
bash: docker: command not found
```

Add Docker-in-Docker to the DevContainer configuration:

```json
"features": {
  "ghcr.io/devcontainers/features/docker-in-docker:2": {}
}
```

Then rebuild the Codespace container.

---

## Git push blocked by Git LFS

If you see:

```text
This repository is configured for Git LFS but 'git-lfs' was not found
```

Install Git LFS:

```bash
apt-get update
apt-get install -y git-lfs
git lfs install
git push origin main
```

---

# Git Ignore Notes

The ROS2 generated folders should stay ignored:

```gitignore
ros2_ws/build/
ros2_ws/install/
ros2_ws/log/
```

If they were already added to Git, remove them from tracking:

```bash
git rm -r --cached ros2_ws/build ros2_ws/install ros2_ws/log
```

Then commit again:

```bash
git add .
git commit -m "Clean ROS2 project structure"
git push origin main
```
